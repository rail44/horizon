# Kitty Keyboard Associated Text

Status: implemented 2026-07-23 (protocol v13).
This is the implementation brief for the first correctness slice from
`horizon-terminal-core`'s resident Kitty keyboard compliance matrix. The
normative protocol is Kitty's
[keyboard protocol](https://sw.kovidgoyal.net/kitty/keyboard-protocol/).

## Problem

Kitty keyboard mode can report both a key's identity and the text generated by
that key. The latter is **associated text**. An application uses key identity
for shortcuts and associated text for insertion, preserving the distinction
across keyboard layouts, dead keys, and composed text.

Horizon currently tracks all five progressive-enhancement flags, including
`REPORT_ASSOCIATED_TEXT`, but its encoder never emits the associated-text
subfield. This is worse than merely not implementing flag 16: a client can set
flags 8 + 16, query the mode successfully, and then receive key events without
the text the accepted mode promises.

The missing value already exists at the UI boundary as GPUI's
`Keystroke::key_char`, but `TerminalCommand::Key` sends only termwiz's
`KeyCode`, `Modifiers`, and Horizon's `KeyEventKind` to sessiond. IME commits
take a separate raw-byte `TerminalCommand::Input` path. Formatting another CSI
sequence inside the UI would therefore duplicate terminal mode state and still
miss keyless composed input; the daemon-owned terminal core remains the only
encoder.

## Decisions

### 1. Keep key identity and generated text separate

Add a Horizon-owned `TerminalKeyInput` payload with these v1 fields:

```rust
pub struct TerminalKeyInput {
    pub key: termwiz::input::KeyCode,
    pub modifiers: termwiz::input::Modifiers,
    pub kind: KeyEventKind,
    pub text: Option<String>,
}
```

`text` is the platform-produced text for this key event, not text reconstructed
from `key`. The UI copies GPUI's `key_char` into it. `None` means that the key
did not generate text or that the platform did not provide it.

The use of termwiz key/modifier types is deliberately retained in this slice.
Replacing them with Horizon-owned key identity, location, lock-state, and
modifier types is the later fix for keypad/base-layout/extended-modifier gaps;
it is not necessary to deliver Associated Text and would turn this bounded
slice into a complete keyboard-input rewrite. The new struct is the expansion
point for optional fields when that work is scheduled.

### 2. Add structured key and text commands; retain legacy commands

Add two terminal commands before `Unknown`:

```rust
TerminalCommand::KeyInput(TerminalKeyInput)
TerminalCommand::TextInput(String)
```

`KeyInput` carries a physical/logical key event and optional generated text.
`TextInput` carries committed text for which no key event is available, most
notably an IME commit. It is semantic text, not pre-encoded PTY bytes.

Keep the existing `Key { ... }` and `Input(Vec<u8>)` variants. They are the
fallback vocabulary for older peers and remain useful for intentionally raw
headless/test input. A new daemon accepts all four:

- legacy `Key` becomes a key input with `text: None`;
- `KeyInput` preserves its text through the core input channel;
- legacy `Input` is written byte-for-byte;
- `TextInput` is encoded according to the terminal's live Kitty mode.

All terminal input encoding remains in `horizon-terminal-core`; neither GPUI
nor the session client inspects or constructs Kitty escape sequences.

### 3. Use protocol v13 as a behavior gate, not a compatibility floor

The new enum variants are additive, but safely choosing them is behavior that
an older daemon does not understand. Bump `SESSION_PROTOCOL_VERSION` to 13 and
add a named `TERMINAL_STRUCTURED_INPUT_VERSION` constant. Keep
`MIN_SUPPORTED_PROTOCOL_VERSION` at 11.

The UI sends `KeyInput`/`TextInput` only when the negotiated version is at least
13. Below 13 it sends the existing `Key`/raw UTF-8 `Input` forms. A v13 daemon
continues accepting old forms from a v11/v12 UI. This gives restart skew a
lossless fallback rather than letting an old daemon discard an unknown input
variant.

The version decision mirrors v12 scroll-windowing: the wire addition is
tolerantly decodable, while the version advertises when the behavior can be
used. Regenerate the committed wire schema after the bump.

### 4. Encode associated text only when both required flags are active

For `KeyInput`, include associated text only when all of these are true:

- `REPORT_ALL_KEYS_AS_ESCAPE_CODES` (flag 8) is active;
- `REPORT_ASSOCIATED_TEXT` (flag 16) is active;
- the event is `Press` or `Repeat`, never `Release`;
- `text` is non-empty and contains no C0/C1 control code.

Encode every Unicode scalar value as decimal and separate multiple codepoints
with `:` in the final CSI-u text field. The encoder must extend its shared CSI-u
builder; it must not append bytes after a completed `u` terminator.

Flag 16 without flag 8 is undefined by the Kitty specification and has no
effect. When either flag is absent, key encoding is byte-for-byte identical to
the current implementation. Associated Text does not repair the separately
recorded shifted-punctuation/base-key deviation: the key field can still be
imperfect while the text field accurately reports what was typed.

### 5. Represent keyless committed text with key code zero

For `TextInput(text)`:

- with flags 8 + 16 and valid non-control text, emit a Kitty CSI-u event with
  Unicode key code `0`, modifier value `1`, and the text codepoints in the
  associated-text field;
- otherwise write the UTF-8 text exactly as Horizon does today.

The raw fallback is intentional for flag 8 without flag 16. A keyless IME
commit has no useful key event to send, and silently losing committed text is a
worse failure than retaining the legacy text behavior.

IME preedit text remains UI-only. Paste remains `TerminalCommand::Paste` and
bracketed-paste behavior is unchanged; a paste is not a keyboard event.
`HORIZON_GPUI_DRIVE` remains raw `Input` because it is a byte-level headless
tap, not platform text input.

### 6. Preserve the existing GPUI/IME deduplication boundary

When report-all-keys is active, GPUI can deliver one printable action through
both `KeyDownEvent` and `EntityInputHandler::replace_text_in_range`. The key
path now sends `KeyInput { text: key_char, ... }`; the text callback still
drops its matching duplicate through `KeyTextDedup`.

A real IME composition commit has no matching key event and therefore reaches
`TextInput`. Direct/ASCII IME mode retains the current recent-key comparison:
if a key event was sent, the duplicate commit is dropped; if the platform
delivered only committed text, it is sent. The phantom-Enter guard and preedit
overlay do not change.

## Data flow

```text
GPUI KeyDown/KeyUp
  -> TerminalKeyInput { key, modifiers, kind, text: key_char }
  -> TerminalCommand::KeyInput (v13) / Key fallback (<v13)
  -> sessiond terminal host
  -> horizon-terminal-core encoder reads live Kitty flags
  -> PTY bytes

GPUI committed text / IME commit
  -> duplicate check
  -> TerminalCommand::TextInput (v13) / Input fallback (<v13)
  -> sessiond terminal host
  -> CSI 0 ... u when flags 8+16, otherwise UTF-8
  -> PTY bytes
```

## Verification

The implementation is complete when all of the following are covered:

1. Encoder unit tests for a plain character, shifted character, non-ASCII
   scalar, multiple codepoints, repeat, release, empty text, and C0/C1 text.
2. Truth-table tests proving associated text appears only for flags 8 + 16 and
   never changes bytes for every other flag combination.
3. Key-code-zero tests for keyless committed text, including the raw UTF-8
   fallback when flag 16 is unavailable.
4. Session-loop/daemon routing tests proving `KeyInput` and `TextInput` reach
   the PTY encoder, while legacy `Key` and `Input` remain unchanged.
5. Negotiated-version tests: v13 uses structured input; v11/v12 use the legacy
   commands without losing text.
6. UI input tests preserving key/text deduplication, IME-only commits, and the
   phantom-Enter guard.
7. Change the resident compliance-matrix Associated Text row from
   `Unimplemented` to `Compliant` and name the enforcing tests.
8. Regenerate the wire-schema artifact, run the full repository gate, build
   the workspace, and run `scripts/check-gpui-terminal.sh`.

## Explicitly outside this slice

- keypad identity and application-keypad handling;
- physical/base-layout key values and complete alternate-key reporting;
- standalone modifier keys, lock-state bits, Hyper/true Meta, and F25-F35;
- the deliberate modified Enter/Tab/Backspace compatibility deviation;
- the pinned alacritty-terminal keyboard-stack query/overflow defects;
- any change to alternate-screen, mouse-mode, or scroll ownership.

Those remain visible in the resident compliance matrix or subsequent terminal
hardening work. Associated Text must not quietly broaden into them.
