# Plan 03 — 役割機構と設定エージェント

Status: ready

## 背景

「レビュー」「モデルルーティング」等の役割つきエージェントを見据えた
土台(プロンプト追加セクション注入点・ツール許可リスト、8e45bb6)は済
み。監査(docs/research/agent-prompting.md Part 2)の結論は「役割の設
計は2つ目の役割が具体化してから」— **設定エージェントがその具体化**。
カラースキームやキーバインドのオンボーディング/設定をエージェントとの
対話で行えるようにする。

## オーナーの留保(設計の前提として重要)

「ドメインエージェント」(Claude でも Horizon でも)が、**エージェント
定義された役割**なのか、**特定のスキルを読み込ませた特化 coder** なの
かは意図的に未決(2026-07-06)。この実装はその分岐の証拠を作る位置づけ
なので、役割機構は最小に留め、スキル機構(agent-prompting Part 3 の最
小形: ファイル規約+必要時読み込み)との役割分担を実装しながら見極める
こと。「定義」に寄せ切る設計を先に固定しない。

## 前提機能(このプランに含む)

- **config の実行時リロード**: 現状 config は起動時読みのみ。設定エー
  ジェントが変更を即反映するには reload 機構が要る。全体リロードか
  theme/keybindings の部分適用かの判断はドメイン内で(Reload Agent
  Runtime コマンドが姉妹の前例)。

## スコープ

- 役割(またはスキル特化)つきエージェントセッションを起動する経路
  (最小: 役割名→追加プロンプトセクション+ツール許可リスト+モデル、
  の対応)。
- 設定エージェント第1弾: カラースキーム([theme]/[theme.ansi])とキー
  バインドの対話的変更 → config.toml への書き込み(fs ツール+承認で足
  りるか専用ツールかはドメイン内判断)→ リロードで反映。
- 将来のカラーピッカービュー(plan なし、roadmap「Later」)との接続は
  設計メモに留める。

## 完了条件

- 設定エージェントを起動し、対話でカラースキーム変更が実際の画面に反映
  される一連が実機で通る。
- 役割/スキルの分担について、実装から得た証拠が設計文書(新設)に記録
  される。
- ゲート緑。config.example.toml と drift guard の同期。
