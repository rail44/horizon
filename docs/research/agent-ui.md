# エージェント出力 UI 調査 — 既存プロダクトの transcript 表現

`docs/plans/application-ui/02-agent-output-ui.md` 段階1の成果物。調査日:
2026-07-06。**この文書は推奨を押し付けるものではない。** 横断的に確認できた
定石と各プロダクト固有の流儀を区別して並べ、最後に Horizon の再設計で判断
が要る論点を「事実 → 選択肢 → 得失」の形で整理する。設計判断は段階2で
オーナーとドメインセッションが行う。

## 対象と情報の確度

| プロダクト | 種別 | 確度 |
|---|---|---|
| Zed (agent panel) | GUI (GPUI) | ソース一次(`crates/agent_ui`、commit `54bf9183`) |

出典 URL は各節に付す。詳細な製品別ノートの主要出典:

Claude Code <https://code.claude.com/docs/en/fullscreen> ほか、
Warp <https://docs.warp.dev/terminal/blocks/> ほか。

## Part 1: 横断して確認できた定石

複数プロダクトが独立に同じ形へ収束しているもの。Horizon が採る場合も
「なぜ効くか」ごと輸入するのが安全で、見た目だけの模倣は避ける。

### 1. ツール呼び出しは「状態+名前+要約」の1行が基本形、詳細は二層目

- ほぼ全製品共通: 折り畳み時も1行目だけで「何が・どうなったか」が読める
  (状態記号/アイコン+ツール名+引数や結果の要約)。
  - Claude Code: `⏺ Bash(cmd)` + 結果先頭3〜4行 + `… +N lines (ctrl+o to expand)`。
  - opencode: ツール種別ごとの記号(Shell=`$`, Edit=`←` 等)+1行、
    折り畳み行数はツール別固定(Shell=10行、Execute=4行、汎用=3行)。
  - Gemini CLI: 状態記号を固定(`✓ o ⊷ ? - x`)+名前太字+1行 truncate。
    コンパクトモードでは常用ツールを `"→ 要約"` や `+N/-N` だけの1行に。
  - Zed: ツール1件=1カード、Disclosure で開閉。edit/terminal 系だけ既定
    展開にする設定あり(`expand_edit_card`/`expand_terminal_card`)。
- 展開の導線は「クリック」と「単一のグローバルキー」の併設が多い
  (Claude Code は ctrl+o とクリック、Gemini CLI は ctrl+o、opencode は
  クリック)。展開キーが複数の場面(ツール出力・MCP 引数・承認プレビュー)
  で同じ1つに統一されているのが共通(Gemini CLI の ctrl+o 流用が典型)。

### 2. 隠した量は必ず数値で可視化する(こっそり切り詰めない)

- `… +N lines` (Claude Code)、`"... first N line(s) hidden ..."`
  (Gemini CLI)、`"…"`+Click to expand(opencode)。
- Zed はさらに「モデルに送った量」まで開示する: ターミナル出力の切り詰めに
  `Truncated` ラベル+「元は X、エージェントには Y だけ送った」ツールチップ。
  人間向け表示の切り詰めと、モデルに渡した内容の切り詰めを区別して見せる。

### 3. thinking/reasoning は本文から視覚的に沈め、既定は畳む方向に収束

- 視覚: 左罫線1本+ミュート色/斜体で「本編ではない」ことを示すのが共通形
  (Zed・opencode・Gemini CLI とも左罫線、opencode は控えめ構文ハイライト
  まで用意)。
- 既定挙動は「ストリーミング中だけ見せて終わったら畳む」に収束:
  - Zed `ThinkingBlockDisplay::Auto`: 最新チャンクだけ自動展開、終われば
    自動で閉じる(手動トグルした場合は尊重)。`Preview`(高さ制限付きで
    見せ続ける)等の4モード。
  - Warp `thinking_display_mode: "show_and_collapse"` が既定。
  - opencode: 既定1行ヘッダ(実行中 `Thinking: <title>`、完了後
    `Thought: <title> · <duration>`)+クリック展開。「1行のまま高さが
    変わらないのでレイアウトがシフトしない」とコメントに明記。
  - 例外は Gemini CLI(思考チャンクごとの独立バブルを常時表示)だが、
    これは Gemini の thought-summary 配信形式に依存した実装。
- API 側の事情: Anthropic 系は thinking の表示形態が `summarized` /
  `omitted` に分かれ、新しいモデルは `omitted`(そもそも流れてこない)が
  既定。「thinking を見せる UI」の価値はプロバイダ・モデル依存で変動する。
  <https://platform.claude.com/docs/en/build-with-claude/extended-thinking>

### 4. 承認 UI は「プレビュー → 質問 → 選択肢」の縦積み+三軸の選択肢

- 構成はほぼ全製品共通: (1) diff/コマンドのプレビュー、(2) 質問文、
  (3) 選択肢リスト。
- 選択肢は「一回だけ許可 / このセッション(または恒久)許可 / 拒否」の
  三軸に収束(opencode・Gemini CLI・Zed(ACP `allow-once`/`allow-always`/
  `reject-once`)・Claude Code)。拒否側に「理由を書いてエージェントに
  返す」導線を持つものが多い(Gemini CLI "No, suggest changes"、opencode
  の RejectPrompt テキストエリア)。
- **承認待ちは強制展開**: Zed は `is_open |= needs_confirmation` を明示的に
  実装(判断材料を隠さない)。Claude Code は auto-scroll をオフにしていて
  も承認プロンプトだけは強制的にビューへスクロールインさせる例外を明記。
- **巨大プレビューが承認ボタンを画面外に押し出さない**: Zed はプレビューを
  高さ制限付きスクロール領域にし、ボタン行は常に見える位置に固定する
  対策をコメント付きで実装。
- 都度ダイアログだけに頼らず、承認の粒度(モード/ルール)を別レイヤーで
  持つのも共通(Claude Code permission mode、Cursor Run Modes、Warp の
  4段階+正規表現 allowlist、Amp の allow/ask/reject ルール)。これは UI
  の見た目より製品思想の話で、Horizon では Phase-1 委任・承認ポリシーの
  論点(cli-control-plane-design.md の Deferred)と接続する。

### 5. 追従スクロールは「明示的な状態機械」として設計する

- 定石(Zed がソースで最も明瞭): 最下部にいる間だけ追従し、ユーザーが
  上へスクロールしたら**明示的な離脱イベント**として記録して追従を止め、
  最下部へ戻ったら再開。Zed は `FollowState::Tail { is_following }` を
  持ち、ターン完了時のスナップとストリーミング中の追従を別々に判定する。
  テストに「キャンセルされた古いターンの stop イベントが新ターンの追従を
  解除してはならない」という不変条件が書かれており、状態管理が自明で
  ないことを示す。
- 復帰導線: 「最下部へ」ボタン/ピル(Cursor 3.0、Claude Desktop の
  改善要求 issue #53382 が求める形)、Zed は加えて「最新のユーザー
  メッセージへ」ボタンを持つ(直近の自分の指示位置=ターン先頭への
  ジャンプ)。
- アンチパターンの実例: Warp は「上へスクロールして読んでいても新規出力が
  押し流す」既定挙動に不満 issue が複数(#3178, #3849, #4066, #8089)、
  Claude Desktop にも同種のバグ報告。読んでいる位置を奪う自動スクロールは
  ユーザーが確実に嫌う、という点は実証済みと言ってよい。

### 6. ターン境界は「ユーザー=箱、アシスタント=地の文」の非対称

- Zed: ユーザーメッセージだけ角丸カード(背景+枠線)、アシスタントは
  枠なしの地の文。区別は色ではなく構造(囲む/囲まない)。
- opencode: ユーザーは左罫線+パネル背景の箱、アシスタントは枠なし+
  ターン末尾にメタ行(モード・モデル名・所要時間、中断時は
  `· interrupted`)。
- Gemini CLI: 行頭記号(`> ` と `✦ `)のみの最軽量形。余白すら「思考→
  応答」「ツール群の境界」という構造的な切れ目にだけ入れる。
- Claude Code(fullscreen)もユーザー発言にだけ背景色トークンを持ち、
  `{`/`}` で前後のユーザープロンプトへジャンプできる=ユーザー発言を
  transcript のアンカー単位として扱う。
- 共通の含意: 「アシスタント出力を吹き出しで囲む」製品は皆無に近い。
  読む量が最も多いアシスタント側を地の文にし、量が少なく再訪の起点に
  なるユーザー側に視覚的な重みを置く。

### 7. diff は「行背景+符号+行番号ガター」、単語粒度があれば尚良し

- 行の背景色で追加/削除、符号(+/-)は別途強調色、行番号ガター付き、が
  共通形(opencode・Gemini CLI・Warp)。Claude Code はさらに単語粒度の
  ハイライト(`diffAddedWord`/`diffRemovedWord`)と、変更行近傍の
  コンテキスト行を減光する `dimmed` トークンを持つ。
- 特例処理も共通: 新規ファイルは diff でなくシンタックスハイライト付きの
  ソース表示(Gemini CLI)、削除ファイルは `-N line(s)` の要約だけ
  (opencode)。ハンク間の省略は罫線で明示(Gemini CLI の二重線)。
- 大きな変更セットは transcript に流し込まず**専用レビュー面に格上げ**が
  GUI 側の潮流: Zed `AgentDiffPane`(複数ファイルを MultiBuffer に集約、
  ハンク単位 keep/reject)、Warp Code Review panel、Amp `⌘;`、Cursor の
  file-level review。transcript 内 diff は要約+導線(`Edits` バー、
  diffstat チップ)に留める。

### 8. 失敗・キャンセルは色+記号+質感変化で即座に判別できる

- 赤系+専用記号(Gemini CLI `x`、opencode は theme.error+権限拒否は
  取り消し線)、Warp はブロック全体を赤背景+赤サイドバー。
- Zed の工夫: 失敗/キャンセルでカードの枠線を実線→**破線**に変える
  (色に頼らない質感の区別)。空のままキャンセルされたツール呼び出しは
  カードごと非表示(「何も生まなかった Canceled カードは無価値」と
  コメントで明言)。

### 9. 長い transcript は仮想化+「可視域外の高さを勝手に変えない」契約

- Zed の transcript は `gpui::List`(可変高対応の仮想化リスト、
  `ListAlignment::Bottom`)。ドキュメントコメントが要求する契約:
  可視領域外の要素が高さを変えるときは `splice`/`reset` で明示通知しな
  ければ壊れる。スクロール位置は「アイテム index+アイテム内オフセット」
  で保持。
- Gemini CLI は確定済み履歴を `<Static>`(再レンダリングしない)へ流し、
  ストリーミング中の部分だけ動かす。長い応答を分割するのも「意味では
  なく描画性能の都合」とコメントに明記。
- Amp は「ちらつきゼロ・ストリーム中もスムーズにスクロール・transcript
  への本物のマウス操作」を独自レンダラ実装の到達目標として掲げた
  (手段は無関係だが、目標水準の参考になる)。
- 含意: 折り畳み既定+明示展開(定石1)はパフォーマンス設計でもある。
  既定で畳まれたブロックは高さが安定し、仮想化・メモ化と相性が良い。

## Part 2: プロダクト固有の流儀(真似る必然のないもの)

- **端末グリフ記法**(Claude Code の `⏺`/`⎿`、opencode の `$`/`←`):
  テキスト端末の制約の産物。ネイティブ GUI はアイコン・インデント・
  カードで同じ機能を表現できる。
- **opencode の `/details` トグル**(成功済みツール呼び出しを transcript
  から丸ごと消す): 情報を大胆に間引く強い判断で、「何が起きたかを後から
  追える」ことを重視するなら合わない。
- **Warp の「1ブロック=1コマンド」メタファー**: ターミナルエミュレータ
  そのものだから成立する。エージェントペインのツール呼び出しは非シェル的
  操作も多く、1:1 では持ち込めない。
- **Zed の承認粒度ドロップダウン**(コマンドパターン単位のチェックボックス
  +Apply)や **ThinkingBlockDisplay 4モード**のような細かい設定分岐:
  成熟後の投資であり、初期実装で揃える必要はない。
- **Gemini CLI のシェルフォーカス切替**(Tab で実行中シェルに直接入力):
  transcript 内に対話シェルを埋め込む特殊機能。Horizon ではターミナル
  ペインが第一級で存在するので、この機構の必要が薄い。
- **Cursor の「承認状態をチャットセッションに紐付ける」設計**: 再起動で
  承認済み diff が復活するバグ群の原因になっており、反面教師。
- **サブエージェントの色分け**(Claude Code の8色、Amp の orb color):
  マルチエージェント機能そのものに紐づくブランディング。必要になった
  ときに Horizon の theme ロールで再設計すればよい。

## Part 3: Horizon への設計論点(事実 → 選択肢 → 得失)

前提となる Horizon の現状資産と制約(plan 02 より):

- transcript は末尾200ブロックのウィンドウ+revision メモ化(b541d7c)。
  セッション長に対する不変コストを回帰させない。
- theme ロール駆動(ハードコード色禁止)。backlog 7: ユーザー吹き出しの
  青と承認バナーの琥珀が中立色に落ちている。
- markdown 描画は `src/agent/view/markdown.rs`、承認バナーは
  `AgentPaneFocus`(フォーカス捕捉の前例)。

### 論点1: ツールブロックの既定表示

- 事実: 全製品が「1行要約+明示展開」の二層開示に収束(定石1・2)。
  折り畳み既定は高さ安定=仮想化・メモ化とも相性が良い(定石9)。
- 選択肢: (a) 全ツールブロックを既定1行に畳む / (b) 種別ごとに既定を
  変える(edit/実行系は数行見せる — Zed 型・opencode 型)/ (c) 現状
  (詳細を流す)を維持し展開だけ整える。
- 得失: (a) は最も読みやすく低コストだが「今なにをしているか」の実況性が
  落ちる(進行中だけ自動展開する Zed の Auto 型で補える)。(b) は
  ツール種別ごとの調整ノブが増える。(c) は読みづらさの主因が残る。
  展開操作は workspace mode のキー判定器との整合(どのキーで展開するか、
  ペイン内フォーカスとの関係)を段階2で決める必要がある。

### 論点2: thinking の見せ方

- 事実: 「ストリーミング中だけ沈んだ見た目で見せ、終わったら1行に畳む」
  が多数派(定石3)。プロバイダによっては thinking がそもそも流れて
  こない方向に動いている。
- 選択肢: (a) opencode 型(1行ヘッダ+クリック展開、レイアウト不動)/
  (b) Zed Auto 型(進行中のみ自動展開)/ (c) 常時表示(現状に近い)。
- 得失: (a) はレイアウトシフトがなく実装も軽い。(b) は「考えている感」の
  実況性が高いが、ストリーム中の高さ変動が仮想化ウィンドウ・追従
  スクロールと絡む。(c) は読み返し時のノイズが最大。

### 論点3: 承認バナー

- 事実: 承認待ちは強制展開+強制ビューイン、プレビューはスクロール領域
  化してボタンを常に見せる、選択肢は三軸(定石4)。Horizon には
  AgentPaneFocus(バナーのキー捕捉)という前例が既にある。
- 選択肢: (a) 現行バナーを維持し、プレビュー(diff/コマンド)を埋め込む
  /(b) ツールブロック内に承認 UI をインライン化(Zed 型)。
- 得失: (a) はフォーカス制御の実装資産を活かせるが、バナーと transcript
  内の該当ブロックの二重表示になりうる。(b) は文脈と判断材料が同じ場所に
  並ぶが、フォーカス捕捉の再設計が要る。いずれでも backlog 7(琥珀の
  復活)はこの論点で回収するのが自然。拒否時に理由を返す導線(定石4)を
  持つかも段階2の判断。

### 論点4: 追従スクロール

- 事実: sticky-bottom+明示離脱+復帰ピルが定石で、押し流しは実証済みの
  アンチパターン(定石5)。Zed は追従状態を明示的な状態機械として持つ。
- 選択肢: (a) Zed 型の追従状態機械+「最下部へ」導線を入れる / (b) 現状
  の挙動を維持。
- 得失: (a) はオーナーの「読みづらい」体感のうち「読んでいる間に流れる」
  成分を直接解消する。実装は 200 ブロックウィンドウ(古いブロックの
  除去でスクロール位置がずれる問題)との整合が必要で、Zed の「index+
  オフセット」方式が参考になる。「最新のユーザーメッセージへ」ジャンプ
  (Zed)はターン再訪の導線として安価に足せる。

### 論点5: ターン境界と話者の視覚化

- 事実: ユーザー=箱/アシスタント=地の文の非対称が GUI・TUI 共通
  (定石6)。アシスタントを吹き出しで囲む製品はない。
- 選択肢: (a) 非対称構造に寄せ、ユーザー吹き出しの青(backlog 7)を
  「箱の背景」ロールとして復活させる / (b) 対称な吹き出し型に戻す。
- 得失: (a) は読む量の多いアシスタント側の幅と密度を最大化でき、
  ユーザー発言がアンカーとして機能する(論点4のジャンプ導線とも接続)。
  (b) を採る根拠は調査からは見つからなかった。

### 論点6: diff・長大出力

- 事実: transcript 内は要約+導線に留め、大差分は専用面へ、が GUI 側の
  潮流(定石7)。ただし専用レビュー面は大きな追加投資。
- 選択肢: (a) 段階2ではまず transcript 内 diff の質を上げる(行背景+
  符号+ガター、新規/削除の特例、diffstat 要約)に絞る / (b) 専用
  レビュー面まで一気に作る。
- 得失: (a) は「読みづらい」の主因(生 diff・長大出力の垂れ流し)を
  低リスクで解消し、専用面は将来の plan に送れる。(b) は Zed/Warp 水準に
  近づくが、workspace のペインモデル(レビュー面をペインにするか
  オーバーレイにするか)という別の設計問題を引き込む。

### 論点7: パフォーマンス制約との整合(横断)

- 事実: 折り畳み状態の変化=ブロック高さの変化であり、仮想化・メモ化と
  追従スクロールの両方に波及する(定石9、Zed の `splice`/`reset` 契約)。
- 含意: 論点1〜6 のどれを採る場合も、「展開/折り畳みは revision を進める
  イベントか」「ウィンドウ外ブロックの状態はどこに持つか」を設計に含める
  こと。既定折り畳み(論点1の (a)/(b))はここでも有利に働く。

# 第2回調査(2026-07-06): crush と GUI エージェント勢

## 経緯と方針

第1回を読んだオーナーとの段階2相談で、実使用の痛点が2つ具体化した:

- **(a) 縦密度が低い** — 返答・ツール呼び出しがそれぞれ別ブロックに
  なり、縦方向の情報量が薄い。
- **(b) edit の意図が読めない** — 出力がツールの output そのままで、
  何をしようとしているのかが分からない。

あわせて設計方針が定まった: **UX の良い先人に倣う(模倣ファースト)**。
独自の発明より、複数製品が収束している定石をそのまま採ることを既定と
し、オーナー判断は製品間で割れている点に絞る。また Floem のネイティブ
GUI なのだから **TUI よりリッチな表現**を狙う。この方針を受けて、TUI で
表現が突出していると評価した crush と、第1回で薄かった GUI エージェント
勢を追加調査した。

追加対象と確度:

| プロダクト | 種別 | 確度 |
|---|---|---|
| crush (charmbracelet) | TUI (Bubble Tea v2/lipgloss) | ソース一次(main、2026-07) |
| Google Antigravity | agent-first IDE | 公式 docs+blog |
| VS Code Copilot (agent mode / Agents window) | IDE | 公式 docs |
| Windsurf Cascade (→Devin Desktop) | IDE | 公式 docs(統合移行中) |
| OpenAI Codex (IDE 拡張 / web / app) | IDE+Web | 公式 docs+コミュニティ |
| Devin / Jules / Conductor / Sculptor | マネージャ型 GUI | 公式 docs+レビュー |
| Kiro / Trae / Amp web / Factory | 参考 | 断片的 |

## Part 4: 縦密度への技法(痛点 a)

複数製品が収束している順に:

1. **要約行1行+折り畳みが基本形**(第1回定石1の再確認+強化)。
   VS Code は `chat.agent.thinking.collapsedTools` でツール詳細を畳み
   **要約行だけを会話に残す**。crush はツールの pending 表示を1行、
   compact 表示(ネスト時)はヘッダ1行のみ。Antigravity は思想として
   「生 tool call を見せず Artifact で検証させる(verification at a
   glance)」と明言する。
2. **連続するツール呼び出しのグルーピング**。Gemini CLI は同一ターンの
   連続ツール呼び出しを1枚の連続カードに束ねる(第1回)。crush は
   `NestedToolContainer` のツリー表示(子は compact 1行)を持つが
   **サブエージェント委譲時にしか使っていない** — この機構を通常の
   ツール列に適用する拡張が Horizon では素直に狙える。
3. **余白・罫線は構造的な切れ目にだけ使う**。crush はターン完了時のみ
   `◇ モデル名 via プロバイダ  所要時間 ────` の水平線フッターを引き、
   ツール呼び出し間には引かない。Gemini CLI は空行すら「思考→応答」
   「ツール群の境界」にだけ入れる(第1回)。
4. **transcript 本体は最小クローム、装飾はモーダル/バッジに予約**。
   crush の「リッチさ」の実体はここにある: transcript の各メッセージは
   左端の縦線1本(フォーカスで太さが変わる)だけで、角丸ボーダーは
   ダイアログ・To-Do/Queue バッジ(pills)にしか使われていない。
   リッチ=装飾を足すことではなく、装飾を置く場所を選ぶこと。
5. **(GUI 定石)俯瞰面を transcript と別サーフェスに分離**。
   VS Code Agents window の Changes パネル(diffstat 一覧)、Antigravity
   の Agent Manager(状態・成果物・承認待ちのダッシュボード)、crush の
   pills(To-Do 進捗の常駐バッジ)。「今なにが変わったか」をスクロール
   ログから独立させることで、ログ側は畳んでも安心して読める。

反面教師(実装が痛点 a を生んでいる実例):

- crush の `list.SetGap(1)` — 全アイテム間に固定1行の空白を機械的に
  挿入しており、リッチと評される crush ですら縦密度はこれで犠牲に
  なっている。Horizon の現状と同型。
- Devin の「小さい表示領域で diff が確認しづらい」批判 — 要約レイヤーを
  挟んでも、詳細への導線が細いと同じ問題が形を変えて残る。
- Warp の追従スクロール押し流し(第1回)。

## Part 5: edit を「意図」として見せる(痛点 b)

1. **生 output を画面に出す経路を持たない**(crush で最も明確)。
   Edit は必ずメタデータ(OldContent/NewContent)から diff レンダラを
   経由し、Write は新規内容のシンタックスハイライト表示、生テキストは
   JSON パース失敗時のフォールバックのみ。**ツールごとに専用レンダラを
   持つ**構造が前提になる。エラー時すら「ERROR タグ+ diff は表示」で、
   何をしようとして失敗したかが残る。
2. **ヘッダは「動詞+対象パス(+diffstat)」**。crush は `● Edit path`
   (diffstat はコピー時のみ付く非対称実装)、VS Code Changes パネルは
   diffstat を明示、Zed の Edits バーは `N files +add/-del`(第1回)。
   複数編集の部分失敗は crush が「Note: X of Y edits succeeded」で明示。
3. **二段階開示(ミニ diff → フル)**。Jules は変更をまずミニ diff で
   縮小表示しクリックで完全エディタに展開。Codex web は task sidebar の
   要約 → review pane。マネージャ型全般が「ライブは要約、1クリックで
   生ログ相当へ降りる」段階的開示に収束している。
4. **特例処理で diff ノイズを削る**: 新規ファイル=ハイライト付きソース
   表示、削除=行数のみ(第1回・Gemini CLI/opencode)、さらに Devin は
   コードの移動/コピーを検出して「移動」として表示し、削除+挿入の
   ノイズを消す。
5. **成果物単位への再構成**(Antigravity)。tool call の列ではなく
   Task List / Implementation Plan / Code Diffs / Walkthrough という
   名前付き Artifact に写像する。ペイン型の Horizon がそのまま採る形では
   ないが、「編集を"ターンの成果"として束ねて見せる」方向の極北として
   参照価値がある。
6. **無いと不満が出る最低線**(Codex IDE 拡張への不満リスト=逆引き
   要求仕様): diff プレビューなしで直接適用される/複数箇所編集で何が
   変わったか分からない/hunk 単位の accept・reject がない/変更箇所への
   ナビゲーションがない。「Antigravity, Copilot, Claude Code, Cursor は
   全部持っているのに」という声まで記録されている。

## Part 6: GUI にしかできない表現(Floem の武器)

TUI 勢が端末ハックで擬似的に実現しているものを、GUI はネイティブに
持てる:

- **sticky header**: スクロール中もツールカードの見出し(何をして
  いるか)を固定表示(Gemini CLI は Ink のフォーク拡張、Warp は Sticky
  Command Header で実現)。Floem では素直に組める。
- **hover での情報開示**: Zed の「Truncated」ラベル+ホバーで元サイズと
  送信サイズを表示(第1回)。行ホバーで「+」コメントボタン(Codex)。
- **インラインコメント**: Antigravity は Artifact に Docs 風コメントを
  残すと、エージェントが実行を止めずに読んで反映する。
- **非テキスト成果物の埋め込み**: Jules は diff 内に生成画像を実画像で
  表示、Antigravity はブラウザ検証の動画を Walkthrough に埋め込む。
- **checkpoint と巻き戻し**: Windsurf/VS Code の名前付きスナップ
  ショット+目次ナビゲーション。ターン境界を「巻き戻せる状態点」として
  扱う。
- **並列の俯瞰**: Devin の Kanban(Running / Waiting for review /
  Done)、Antigravity Manager のカード一覧。Horizon では workspace/
  セッション管理(plan 01 のモーダル)との接続先。
- **質感による状態表現**: Zed の実線→破線(失敗/キャンセル)、crush の
  シマー/グラデーション(意匠として抽出し、実装は GUI のアニメーション
  API へ)。

ただし装飾の総量は Part 4-4 の原則(transcript は最小クローム)と両立
させる — リッチにするのは表現力であって、飾りの量ではない。

## Part 7: 論点の更新(第1回の論点への反映)

模倣ファースト方針の帰結として、第1回の論点のいくつかは「定石をそのまま
採る」で実質決着し、オーナー判断が要る点は絞られる:

- **論点1(ツールブロック)— 具体形が定まった**: 既定は「状態+動詞+
  対象(+結果要約)」の1行。進行中のみ自動展開(または進行表示)、
  クリック/キーで展開。連続ツール呼び出しはグルーピングして束ねる。
  ここは全製品が同じ方向で、判断余地はグルーピングの粒度程度。
- **論点6(diff)— 優先順位が上がった**: 「edit 専用レンダラ(生 raw の
  経路を持たない)+動詞+パスのヘッダ+新規/削除の特例」は痛点 (b) の
  直接解であり、専用レビュー面(大きい投資)より先に着手する価値がある。
- **新論点8: ターン境界のフッター** — crush の「ターン完了時のみ水平線+
  モデル名+所要時間」は、縦密度と両立するターン境界表現として第1回の
  論点5を具体化する。採るかどうかとフッターに載せる情報(モデル・所要
  時間・中断表示)が判断点。
- **新論点9: 俯瞰面** — Changes/Todo の集約パネルを v1 に含めるか。
  GUI 定石だが追加投資であり、transcript 内の改善(論点1・6)を先に
  着地させてから判断でよい素材が揃った。
- **残るオーナー判断**: 製品間で本当に割れているのは (i) thinking の
  見せ方(第1回論点2の3案)、(ii) 俯瞰面の有無とタイミング(新論点9)、
  (iii) 承認 UI の置き場所(第1回論点3: バナー維持かインライン化か)。
  それ以外は定石追従で進められる。
